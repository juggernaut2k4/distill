# CONTENT-POP-01 — Live-Conductor Content Population Fix + Provisioning Self-Heal — Requirement Document
Version: 1.0
Status: DRAFT
Author: Business Analyst Agent
Date: 2026-07-05

---

## 1. Purpose

Sessions using the newer "live-conductor" content pipeline reach the live call with an empty
knowledge base and empty script, even though the content was successfully generated by the
background pipeline and is sitting in the database (`sessions.live_conductor_content`). This is
invisible at launch time because the guard that would normally catch "no content" is explicitly
bypassed for this path — the code checks a flag that says content exists (correctly — it does),
but then never reads that content into the fields (`freshSections` / `trainingScripts`) that the
rest of the route, and downstream provisioning, actually use. Clio ends up on a live call with a
participant and nothing to teach from.

A second, compounding failure rides on the same gap: because `topic_title` is one of the fields
that can go unpopulated on this path today, two webhook handlers snapshot the literal string
`'Unknown Session'` into the user's `sessionHistory[]` at call-end. That polluted string is
re-injected into every future session's prompt under "RECENT SESSIONS" — so a single failed
session's damage compounds forward across all future sessions for that user, not just the one
that failed.

Without this fix: live-conductor sessions silently degrade to a content-less call (the worst
possible failure mode — indistinguishable from success until the participant notices Clio has
nothing to say), and session history keeps accumulating corrupted entries that poison future
context.

This spec covers two parts: Part A fixes the specific, confirmed root cause. Part B adds a
defensive, self-healing pre-flight check at the point Hume actually gets provisioned, so any
*future* cause of empty content (a regression, a race condition, a partial write, a new bug) also
self-corrects rather than silently degrading the call.

---

## 2. User Story

As an executive using Clio for a scheduled live coaching session on the live-conductor pipeline,
I want the session to always launch with real, generated content already loaded,
So that Clio can actually teach the material I was scheduled to learn, and my session history
stays accurate for future sessions.

As the on-call engineer/Arun monitoring production,
I want any case of empty content reaching provisioning to trigger an automatic, logged recovery
attempt instead of a silent bad call,
So that failures are self-healing where possible and loudly diagnosable where not.

## 6. Data Requirements

### Part A — exact field mapping: `live_conductor_content.tabs[]` → `freshSections` / `trainingScripts`

Source shape, confirmed from `lib/content/live-conductor-content.ts` lines 27-37 (`LiveConductorTab`
within `LiveConductorContent.tabs[]`) and `lib/content/session-content-generator.ts` lines 64-84
(`ContentArticle`, the `article` field's type):

```
LiveConductorTab {

  subtopic_title: string
  article: ContentArticle {

    subtopic_slug: string

      overview: string
      key_facts: string[]
      how_it_works: string

      common_misconceptions: string[]

    industry_angle: string
  }
}
```

Target shapes, confirmed from `lib/templates/types.ts` lines 390-413 (`TemplateSection` discriminated
union) and `lib/clio-context-builder.ts` lines 30-34 (`TrainingScript`, the shape
`buildSessionScript`/`buildTopicContext` already consume — this is the actual contract, not the
DB-row `training_script` JSONB shape written in `session-content-pipeline.ts` lines 417-424, which
additionally carries `visualization_spec` that `TrainingScript` here does not need):

```
TemplateSection (discriminated union member used here: type = 'DefinitionTriptych')
{
  id: string
  type: 'DefinitionTriptych'
  data: DefinitionTriptychData { term, category, what_it_is, real_example{company,what,result}, common_myth, so_what }
  meta: TemplateMeta { subtopicTitle, sessionTitle, userRole, userIndustry }
  status: SectionStatus
}

TrainingScript {
  subtopic_title: string
  subtopic_slug: string
  segments: TrainingSegment[]   // TrainingSegment = { type: 'TEACH'|'CHECKPOINT'|..., content: string, variants?: CheckpointVariants }
}
```

**Field-by-field mapping table** (one row of `tabs[]` → one `TemplateSection` + one `TrainingScript`):

| Source field (`LiveConductorTab` / `ContentArticle`) | Target field | Target object | Notes |
|---|---|---|---|
| `tab.subtopic_slug` | `id` | `TemplateSection` | Direct copy |
| `'DefinitionTriptych'` (literal) | `type` | `TemplateSection` | Fixed choice — see rationale below |
| `tab.article.subtopic_title` | `data.term` | `TemplateSection` (`DefinitionTriptychData`) | Direct copy |
| `'AI Concept'` (literal constant) | `data.category` | `TemplateSection` | No category field exists in `ContentArticle` — use a fixed literal; this is a technical/display-only value, full autonomy per `CLAUDE.md` |
| `tab.article.sections.overview` | `data.what_it_is` | `TemplateSection` | Direct copy — `overview` is already 100-150 words plain-English, matching `what_it_is`'s doc comment |
| `'Illustrative example'` (literal) | `data.real_example.company` | `TemplateSection` | `ContentArticle.illustrative_example` is prose, not `{company, what, result}` — see mapping below |
| `tab.article.sections.illustrative_example` | `data.real_example.what` | `TemplateSection` | Direct copy of the full prose string into `.what` |
| `''` (empty string) | `data.real_example.result` | `TemplateSection` | No separate "result" field exists in `ContentArticle` — `illustrative_example` is a single prose block; leave `.result` empty rather than fabricating a fact not present in source data (see Section 9 edge case) |
| `tab.article.sections.common_misconceptions[0]` (first entry, or `''` if empty) | `data.common_myth` | `TemplateSection` | `ContentArticle` gives an array of 2-4 myths; `DefinitionTriptychData.common_myth` takes one string — use the first entry |
| `tab.article.sections.enterprise_implications` | `data.so_what` | `TemplateSection` | Direct copy — closest existing field to a "so what" framing |
| `tab.article.subtopic_title` | `meta.subtopicTitle` | `TemplateSection` | Direct copy |
| `sessionTitle` (already in scope in `recall/bot/route.ts`, line 83) | `meta.sessionTitle` | `TemplateSection` | Direct copy from existing local variable |
| `userRole` (already in scope, line 91) | `meta.userRole` | `TemplateSection` | Direct copy from existing local variable |
| `userIndustry` (already in scope, line 92) | `meta.userIndustry` | `TemplateSection` | Direct copy from existing local variable |
| `'ready'` (literal) | `status` | `TemplateSection` | Matches the `SectionStatus` value already used for the synthetic overview section (line 316) |
| `tab.article.subtopic_title` | `subtopic_title` | `TrainingScript` | Direct copy |
| `tab.article.subtopic_slug` | `subtopic_slug` | `TrainingScript` | Direct copy |
| *(see below)* | `segments` | `TrainingScript` | See segment mapping below |

**Segment mapping** — `ContentArticle` has no script/segments at all (confirmed: no `segments`,
`TEACH`, `CHECKPOINT` fields anywhere in the `ContentArticle` interface,
`session-content-generator.ts` lines 64-84). The live-conductor path is explicitly script-less by
design (per the file header comment in `live-conductor-content.ts` line 6: "script-less live
conductor path"). Therefore each `TrainingScript.segments` array is built as a single synthetic
`TEACH` segment so `buildSessionScript` (which expects to find a `'TEACH'` segment via
`script?.segments.find(s => s.type === 'TEACH')`, `clio-context-builder.ts` line 146) has real
content to surface instead of falling back to its own generic placeholder text
(`(No script — explain the key concepts from the knowledge base in plain language.)`, line 175):

```
segments: [
  {
    type: 'TEACH',
    content: `${tab.article.sections.overview} ${tab.article.sections.how_it_works} ${tab.article.sections.enterprise_implications}`
  }
]
```

No `CHECKPOINT`/`PROBE`/`CONTINUE` segments are synthesized — `buildSessionScript` already falls
back gracefully to generic prompts for those (`How does that land for you?`, `Let me try a
different angle.`, `Good. Let's move to the next section.` — lines 178, 182, 187) when absent, and
inventing checkpoint questions/variants not present in the source `ContentArticle` would risk
contradicting the actual didactic content, per the "never invent UX/content the brief didn't
specify" principle applied here to generated-content fidelity.

**Rationale for fixed `type: 'DefinitionTriptych'`**: `selectTemplate()` (`lib/templates/selector.ts`)
picks a template based on subtopic title keywords and an LLM-provided `template_hint` — neither
input exists on this path (no `visual_spec.template_hint` field in `ContentArticle`). Rather than
invoke `selectTemplate`/`generateTemplateData` again (out of scope — Known Constraint says "no
change to how content is normally generated" and this is a read-time mapping, not a generation
call), this mapping uses a single fixed, generically-appropriate template for every live-conductor
tab. `DefinitionTriptych` was chosen because its shape (`what_it_is` / `real_example` /
`common_myth` / `so_what`) is the closest existing structural match to the fields
`ContentArticle.sections` actually provides, of all 22 template types in `TemplateName`
(`lib/templates/types.ts` lines 363-386).

### Part B — exact "suspiciously empty" check

Evaluated at `app/api/hume-native/provision-config/route.ts`, immediately after reading
`stateRow.topic_title` / `stateRow.sections` / `stateRow.training_scripts` (existing lines 70-82).
Content is considered **empty** if ANY of the following is true:

1. `stateRow.sections` is `null`, `undefined`, or an array with `.length === 0`.
2. Every element of `stateRow.sections` (when non-empty) has no usable text — defined precisely as:
   for each section, extract the first present, non-empty string among
   `data.what_it_is`, `data.overview` (if present on that template's data shape),
   `data.direct_answer`, `data.one_line`, or (universal fallback) `JSON.stringify(section.data)`;
   if that extracted string's `.trim().length < 40` characters for every section, treat the whole
   set as empty. (40 chars ≈ shorter than any real generated sentence — a deliberately generous
   floor to avoid false positives, not a content-quality check.)
3. `stateRow.topic_title` is `null`, `undefined`, or `.trim().length === 0`.
4. The composed `sessionContent` string (existing lines 91-95, already computed today) is under
   200 characters total after trimming. This is a cheap, single top-level check that also catches
   any case not caught by 1-3 (e.g. `buildTopicContext`/`buildSessionScript` both degrading to
   their own internal fallback text for every section — which happens if `sections` is non-empty
   but genuinely content-less).

Any single one of these four conditions being true triggers the self-heal flow. This check runs
**before** `topicContextDocs` is constructed (before existing line 89) so the self-heal can
replace `sections`/`trainingScripts` before anything downstream reads them.

### Part B — exact self-heal flow

1. Log detection (see logging spec below).
2. Re-derive inputs for `generateLiveConductorContent(sessionId, topicId, topicTitle,
   subtopicTitles, userId, userContext)` — confirmed callable, see below:
   - `sessionId` — already resolved in this route at line 66, no new query needed.
   - `userId` — already available directly from the request body at line 39, no new query needed.
   - `topicId` — per the established convention in `inngest/session-content-pipeline.ts` line 163
     ("Always key content by DB session UUID"), `topicId = sessionId`.
   - `topicTitle` — fetch `sessions.session_title` for this `sessionId` (falls back to
     `'AI Strategy Session'` if null, matching `session-content-pipeline.ts` line 164's exact
     fallback for consistency).
   - `subtopicTitles` — fetch `sessions.session_plan`, `sessions.sub_sessions`, `sessions.topics`
     for this `sessionId` and apply the exact same 3-tier priority logic already used at
     `session-content-pipeline.ts` lines 167-179 (session_plan.sub_sessions →
     session.sub_sessions JSONB → `getSubtopicsForSession` fallback). This requires either
     importing that exact resolution logic or duplicating the same three-line priority chain
     inline — implementer's choice, but the resulting priority order and fallback values must be
     identical to `session-content-pipeline.ts`, not reinvented.
   - `userContext` — fetch `users.role, industry, ai_maturity, role_level` for this `userId`
     (already have `userId`) and apply the exact same shape/fallback as
     `session-content-pipeline.ts` lines 181-186 (`role ?? 'executive'`, `industry ?? 'business'`,
     `maturity: ai_maturity ?? 'beginner'`, `roleLevel: role_level ?? inferRoleLevel(role)` — the
     `inferRoleLevel` helper is a private function in `session-content-pipeline.ts` lines 41-48;
     since it is not exported, either export it from that file or reimplement the identical
     4-branch regex logic inline. This is a small technical decision (export vs. duplicate a
     10-line pure function) — full autonomy per `CLAUDE.md`, no CEO input needed.
3. Call `generateLiveConductorContent(sessionId, topicId, topicTitle, subtopicTitles, userId,
   userContext)` (imported from `@/lib/content/live-conductor-content` — exported, confirmed
   directly by reading the file, lines 205-229) — **await it synchronously**, no `step.run`, no
   Inngest event. Confirmed safe to call from a plain API route: this function and everything it
   calls (`buildLiveConductorTabs` → `generateContentArticles`; `generateTopicBackground`) contain
   zero references to Inngest's `step` parameter anywhere in `lib/content/live-conductor-content.ts`
   or `lib/content/session-content-generator.ts` (confirmed by direct code read of both files in
   full) — they are plain `async function`s using only the Supabase admin client and the
   Anthropic SDK, exactly like every other `lib/content/*` function already called synchronously
   elsewhere in the codebase (e.g. `generateTopicContextDoc`, already called synchronously inside
   this same route's sibling, `recall/bot/route.ts` line 206). Also confirmed: `live-conductor-
   content.ts` contains zero DB writes of its own (no `supabase`, `.update(`, `.upsert(` anywhere
   in that file) — it is a pure compute function; the caller owns all persistence, so calling it
   a second time here cannot double-write anything by itself.
4. **Timeout**: wrap the call in a 60-second timeout (`Promise.race` against a timer, or an
   `AbortController`-based pattern if the Anthropic SDK call supports it directly — implementer's
   choice of mechanism). 60 seconds is chosen as a generous ceiling for one whole-topic background
   generation (`generateTopicBackground`, capped at `max_tokens: 4000`,
   `live-conductor-content.ts` line 132) plus N per-subtopic `ContentArticle` generations
   (typically 3-5 subtopics per session) run via `generateContentArticles`. This value is a
   technical/config decision under full autonomy per `CLAUDE.md` — no further confirmation needed.
   `provision-config/route.ts` currently has no `export const maxDuration` — one must be added
   (120, matching the existing precedent set in `app/api/recall/bot/route.ts` line 22) so the
   route itself is not killed by the platform before the 60s self-heal timeout can even fire and
   be handled gracefully.
5. **Retry policy**: exactly one attempt, no retry-on-failure for the generation call itself (the
   call to `generateLiveConductorContent` either succeeds within the timeout or it doesn't — no
   second LLM-call attempt). This matches Arun's stated expectation in the brief ("try once, if it
   fails or times out, block the call"). However, per Section 4 step 4d above, there IS one
   re-check of the *completeness check* (not a second generation attempt) after a successful
   generation call, to confirm the newly generated content actually satisfies the same emptiness
   check from Section 6 above before proceeding — this is a correctness verification step, not a
   retry of generation.
6. On success (generation completes within timeout AND the post-generation completeness re-check
   passes): write `sessions.live_conductor_content` + `content_status: 'ready'` for this
   `sessionId` (mirroring `session-content-pipeline.ts` lines 234-239's exact write shape), and
   update the local `sections`/`trainingScripts`/`sessionContent` variables in this request's
   execution using Part A's mapping logic (Section 6 above) so provisioning proceeds with the
   fresh data in the same request — no second round-trip to `walkthrough_state` is required since
   the mapped data is already in memory.
7. On failure (throws, times out, or the post-generation re-check still shows empty): return
   `NextResponse.json({ error: 'Session content unavailable — on-demand generation failed' },
   { status: 502 })` — same status code convention the route already uses for its other hard
   failure (`provisionNativeConfig` failure, existing line 119).

### Part B — exact logging

All logs use the existing `[hume-native/provision-config]` prefix convention already used
throughout this file (e.g. lines 63, 77, 118, 134).

- **On emptiness detected**: `console.warn('[hume-native/provision-config] CONTENT-POP-01: empty content detected for session', sessionId, '— triggering on-demand generation', { sectionsCount: sections.length, topicTitle: stateRow.topic_title, sessionContentLength: sessionContent.length })`
- **On self-heal generation starting**: `console.log('[hume-native/provision-config] CONTENT-POP-01: starting synchronous on-demand generation for session', sessionId)`
- **On self-heal success**: `console.log('[hume-native/provision-config] CONTENT-POP-01: on-demand generation succeeded for session', sessionId, '— proceeding with fresh content', { tabsGenerated: <n>, durationMs: <elapsed> })`
- **On self-heal timeout**: `console.error('[hume-native/provision-config] CONTENT-POP-01: on-demand generation TIMED OUT after 60s for session', sessionId)`
- **On self-heal generation throwing**: `console.error('[hume-native/provision-config] CONTENT-POP-01: on-demand generation FAILED for session', sessionId, err instanceof Error ? err.message : err)`
- **On post-generation re-check still empty**: `console.error('[hume-native/provision-config] CONTENT-POP-01: on-demand generation completed but content is STILL empty for session', sessionId, '— blocking call')`

All logs include `sessionId` so they are greppable in Vercel runtime logs without DB access, per
the brief's explicit requirement. No secrets (API keys, tokens) are ever included in any of the
above, consistent with `CLAUDE.md`'s secrets-handling rule.

## 8. Error States

**Part A:**
- If `liveConductorContent.live_conductor_content.tabs` contains a malformed entry (e.g. missing
  `article.sections` fields), the mapping in Section 6 does not throw — every field read uses the
  optional-chaining / fallback pattern already established elsewhere in this file (e.g.
  `sessionData?.session_title ?? 'AI Coaching Session'`), so a malformed tab produces a section
  with empty-string fields rather than crashing the route. This is consistent with the existing
  file's defensive style throughout.
- If the Supabase read of `sessionData` itself fails (pre-existing, unrelated to this fix): no
  change — existing top-level `try/catch` around the whole route body (line 52 onward) already
  returns HTTP 500 with `{ error: 'Failed to create bot' }` (line 368), unchanged.

**Part B:**
- Self-heal generation throws → HTTP 502, logged, no content sent to Hume (Section 7, acceptance
  test 6).
- Self-heal generation times out → HTTP 502, logged, no content sent to Hume (Section 7, test 7).
- Self-heal succeeds but re-check still fails → HTTP 502, logged (Section 7, test 8).
- If the re-derivation queries (for `topicTitle`/`subtopicTitles`/`userContext`) themselves fail
  (e.g. the `sessions` or `users` row is missing) — treat identically to a generation failure:
  log with the same `[hume-native/provision-config] CONTENT-POP-01` prefix noting which
  re-derivation step failed, and return HTTP 502. Do not attempt generation with missing/`undefined`
  inputs.
- No loading-state UI change is in scope (Section 4) — the existing bot-connecting UI already
  covers this wait with its current loading indicator; the only change is its duration.

## 9. Edge Cases

- **Empty `common_misconceptions` array** in a `ContentArticle` (theoretically possible if the LLM
  under-delivers) → `data.common_myth` maps to `''` (empty string) per the mapping table's explicit
  fallback, not a thrown error.
- **`illustrative_example` is itself empty/very short** → `data.real_example.what` will be short;
  this is a data-quality issue upstream in content generation (out of scope for this fix — this
  fix only maps whatever exists, it does not validate the generation quality of `ContentArticle`
  itself, which is a separate, already-shipped pipeline).
- **A session has `hasLiveConductorContent: true` but zero tabs somehow slip past the truthiness
  check** — cannot occur: `hasLiveConductorContent` already requires
  `live_conductor_content?.tabs?.length` truthy (existing line 278), so the new mapping code always
  receives a non-empty `tabs` array by construction.
  it existing.
- **Part B's self-heal races with the background pipeline re-running for the same session** (e.g.
  a webhook or retry re-triggers `session-content-pipeline.ts` at the same moment provisioning
  detects emptiness) — both writers target the same `sessions.live_conductor_content` column with
  a simple `.update()` (last-write-wins, no optimistic locking in either path, matching the
  existing pipeline's own concurrency model, which already has no such protection today). This is
  an accepted, pre-existing risk pattern in this codebase, not a new one introduced by this spec —
  no new locking mechanism is in scope here (would be a separate, larger spec if desired).
- **First live-conductor session for a user vs. a returning user** — no different handling; the
  mapping and self-heal logic is identical regardless of session history.
- **Self-heal triggers for a session that is NOT on the live-conductor pipeline at all** (i.e. an
  old-style session whose `walkthrough_state.sections` happens to be empty for a legitimate,
  different reason) — Part B's self-heal always calls `generateLiveConductorContent`, which
  produces live-conductor-shaped content regardless of which pipeline the session was originally
  on. This is accepted as-designed per the brief: Part B is a generic backstop for "empty content
  reaching provisioning," not specifically scoped to live-conductor sessions only — `provision-
  config/route.ts` itself is only ever invoked when `NEXT_PUBLIC_HUME_NATIVE_ENABLED` is true for
  that session (per its own header comment, lines 17-18), and the self-heal's job is simply "make
  sure there's real content before this specific provisioning call," regardless of origin pipeline.
- **Mobile vs. desktop** — not applicable; this is a server-side/API fix with no client rendering
  differences.
- **Slow network / API timeout** — covered by the 60s timeout in Section 6, Part B.

  `docs/specs/HUME-NATIVE-01-phase-c-nightly-cleanup-feature-brief.md` /
  `...-requirement-doc.md`). Not touched by this spec.

  adds an on-demand synchronous trigger as a fallback path inside `provision-config/route.ts`.

  from before this fix ships. See the explicit, separate, optional recommendation in a note below.
- Any UI-facing change of any kind.
- Any change to `recall/bot/route.ts`'s audit-token logic (lines 68-81, 330 walkthrough_state
  read/write of `audit_token`) — completely untouched, not referenced anywhere in this spec's
  changes.
- Any change to `TemplateSection`/`TrainingScript`/`ContentArticle`/`LiveConductorContent` type
  definitions themselves — this spec only maps between existing shapes, it does not modify any
  type.
- Improving the quality, accuracy, or richness of `ContentArticle` generation itself (e.g. filling
  in a more meaningful `real_example.company`/`.result` than empty strings) — that would require
  changing the upstream `ContentArticle` shape or `generateContentArticles`' prompt, which is
  explicitly out of scope per the brief's Known Constraints ("no change to how content is normally
  generated").
- Adding optimistic locking / concurrency protection between the self-heal write and the
  background pipeline's write to the same `sessions` row (see Section 9 edge case) — flagged as a
  possible separate follow-up, not built here.
- Choosing a different template type per live-conductor tab (e.g. running `selectTemplate` +
  `generateTemplateData` again) instead of the fixed `DefinitionTriptych` choice — would require an
  additional LLM call per tab at read-time, which is a generation-pipeline change, out of scope.

# Design: Session Title Consistency — Single Source of Truth

Generated by /office-hours on 2026-06-10
Branch: main
Repo: distill

---

## Problem Statement

Multiple agents generate titles for the same session independently. The curriculum engine sets a title in `curriculum_plans.visible_sessions[].title`. The session designer generates a different title and stores it in `sessions.session_title`. The user sees both in the UI — the plan screen and sessions screen show the plan title as the group header, the session row shows the designer's title. These are different strings for the same thing. The user is confused about which title is the real one.

**Current broken flow:**
```
Curriculum Engine → visible_sessions[].title = "Introducing Claude for Work — Why It Matters for a Technology Leader"
Session Designer  → sessions.session_title   = "Claude in Financial Services: Safety Architecture, Deployment Models, and C-Suite Value for Technology Leaders"
UI (plan screen)  → reads visible_sessions[].title  ← different
UI (session row)  → reads sessions.session_title     ← different
```

Two titles. One session. User confusion.

---

## Decision: Option B — Curriculum Plan Title is Canonical

**Rule:** The curriculum engine sets the session title once. Every other agent reads from it. No agent downstream of the curriculum engine may generate, override, or rename a session title.

```
Curriculum Engine → sets visible_sessions[].title (once, final)
Session Designer  → reads visible_sessions[].title → stores verbatim as sessions.session_title
KB generator      → reads sessions.session_title (same string)
UI everywhere     → reads sessions.session_title (same string everywhere)
```

One title. One source. Zero drift.

---

## What This Means in Practice

### What is LOCKED (do not change)
- Topic generation — unchanged
- Arc structure — unchanged  
- Curriculum plan visible_sessions shape — unchanged
- Session designer subtopic generation — unchanged
- Number of sessions, depth, sequencing — all unchanged

### What CHANGES

**1. Curriculum engine must generate the final, role-specific title — not a generic placeholder**

The curriculum engine already knows: role, industry, ai_maturity, and the session's focus. It must use these to generate a title that is specific enough to be the permanent title. No "Introducing X" placeholders that the session designer is expected to improve.

Good title (specific, role-tailored):
> "Claude in Financial Services: Safety Architecture, Deployment Models, and C-Suite Value for Technology Leaders"

Acceptable but generic (current state — needs to improve):
> "Introducing Claude for Work — Why It Matters for a Technology Leader"

**2. Session designer must NOT generate its own title**

The session designer receives `title` from `visible_sessions` and stores it verbatim in `sessions.session_title`. It contributes: subtopics, duration, depth breakdown. It does not contribute: the session title.

**3. All UI reads from `sessions.session_title` as the single source**

- Plan screen session row → `sessions.session_title`
- Sessions tab group header → `sessions.session_title` (not `visible_sessions[].title`)
- Sessions tab session row → `sessions.session_title`
- KB index entry → `sessions.session_title`
- KB arc overview card → `sessions.session_title`
- Email/SMS notifications → `sessions.session_title`
- Google Calendar invite title → `sessions.session_title`

If `sessions.session_title` is null (not yet designed), fall back to `visible_sessions[].title`. Never display both.

## Title Hierarchy — The Full Canonical Chain

```
Arc title      → curriculum_plans.visible_sessions[].arc_name
                 (e.g. "Anthropic Claude for Work")
                 Source of truth: curriculum engine
                 Used: everywhere an arc name appears

Session title  → sessions.session_title
                 (e.g. "Claude in Financial Services: Safety Architecture...")
                 Source of truth: curriculum engine → written verbatim by session designer
                 Used: everywhere a session name appears

Subtopic title → topic_content_cache.subtopic_title
                 (e.g. "Constitutional AI and Enterprise-Grade Safety: What Sets Claude Apart")
                 Source of truth: session designer (owns subtopics)
                 Used: everywhere a subtopic name appears
```

Each level has exactly one owner. No level can be renamed by a downstream agent.

---

## Constraints

- No migration needed — session designer can simply start inheriting the title
- Curriculum engine title improvement is a prompt change, not a schema change
- The plan screen's group header currently reads `visible_sessions[].title` — it should switch to reading `sessions.session_title` once available (with fallback)
- For the 8 currently-scheduled SCH-01 shell sessions (idx 2, 4–10): these have `curriculum_session_id: null` and custom titles. When session designer runs for them and sets `curriculum_session_id`, the title should also be set from the plan. These are a one-time cleanup.

---

## Success Criteria

- ✓ Plan screen and sessions screen show identical titles for the same session
- ✓ KB index, arc overview, and session header all show the same title
- ✓ No agent in the pipeline generates a second title for a session that already has one
- ✓ A new user going through onboarding → plan → sessions sees one consistent title at every step
- ✓ The curriculum engine's generated title is specific enough to serve as the permanent title (no vague "Introducing X" placeholders)

---

## What Is NOT In Scope

- Changing how topics are generated or selected
- Changing the arc structure or session count
- Changing how subtopics are generated (session designer still owns subtopics)
- Changing session duration or depth logic
- Any UI redesign beyond reading from the correct field
